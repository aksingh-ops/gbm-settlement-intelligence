Trade settlement fail rate monitoring, counterparty risk ranking, and SLO breach prediction for Global Banking and Markets (GBM) operations analytics. Built around the SEC's publicly available Fails-to-Deliver dataset and the regulatory framework of SEC Regulation SHO Rules 203 and 204.

The project addresses the core GBM Operations problem: identifying settlement failures before they breach regulatory thresholds, attributing them to specific counterparties and operational causes, and forecasting future fail volumes to enable proactive resource management.

---

## Business Context

The US equity market transitioned to **T+1 settlement on May 28, 2024** (SIFMA/ICI/DTCC T+1 After Action Report). This compressed the exception resolution window from two days to one day, increasing the stakes for early-warning analytics.

The SIFMA/DTCC/ICI **T+1 After Action Report (September 2024)** confirmed that post-T+1 CNS fail rates held at **2.12%** — consistent with T+2 era benchmarks. The Ontario Securities Commission's **"Impact of T+1 Settlement on Failed Trades" (2025)** further validated that ETF and non-ETF fail rates remained within historical ranges.

**Regulatory anchor:** SEC Regulation SHO Rule 204 requires mandatory close-out of:
- Short sale fails by **T+4**
- Long sale fails by **T+6**

Breaching these thresholds triggers regulatory reporting requirements. This project's SLO monitoring layer uses these deadlines as the Green / Yellow / Red status thresholds.

## How to Run

```bash
git clone https://github.com/aksingh-ops/gbm-settlement-intelligence
cd gbm-settlement-intelligence
pip install -r requirements.txt

# Run with synthetic data (parametrized from DTCC/SEC/OSC published benchmarks)
python run_pipeline.py

# Run with real SEC data
# 1. Download from: https://www.sec.gov/data-research/sec-markets-data/fails-deliver-data
# 2. Place cnsfails{YYYYMM}{a|b}.zip files in data/raw/
python run_pipeline.py --real
```

One command produces all 9 outputs in `reports/`.

## Data Sources and References

This project uses synthetic data parametrized from the following published sources. The `--real` flag substitutes real SEC data without any code changes.

| Source | Use in project |
|---|---|
| SIFMA/ICI/DTCC T+1 After Action Report (Sept 2024) | CNS fail rate benchmark (2.12%), T+1 transition date, non-CNS benchmark (3.31%) |
| OSC "Impact of T+1 Settlement on Failed Trades" (2025) | Fail rate distribution by security type, statistical testing methodology |
| SEC Regulation SHO Rule 204 | SLO threshold definitions (T+4 for short sales) |
| SEC Rule 203(c)(6) | Threshold security definition (10,000 shares, 5 consecutive days) |
| NY Fed Liberty Street Economics, "Measuring Settlement Fails" (2014) | DTCC CNS data methodology and historical context |
| SEC Fails-to-Deliver Data | Primary real data source (public, updated twice monthly) |
| FINRA OTC Threshold Securities | Daily threshold security list |
| Jayson, D. "Improving Trade Settlement Fail Prediction with AI/ML" (LinkedIn, 2018) | Industry precedent for ML application to settlement fails |

## Limitations

- **Synthetic data for development:** The development dataset is generated from published DTCC/SEC/OSC parameters, not from direct DTCC participant-level data. Use `--real` with SEC public files for production validation.
- **Aggregated SEC data:** The SEC public dataset reports aggregate daily fails per security, not individual trade-level counterparty records. Counterparty granularity in this project is synthetic.
- **SARIMAX MAPE 16.4%:** Settlement fail rates are inherently noisy. A 16.4% MAPE reflects the difficulty of forecasting a high-variance operational metric, not a model failure.
- **XGBoost optimized for recall:** The model prioritizes catching actual breaches (recall 0.694) over minimizing false alerts (precision 0.155). This is the correct tradeoff for an early-warning system.
